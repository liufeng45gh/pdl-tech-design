### 编码规范

#### 1.1为什么需要编码规范
```
编码规范对于程序员而言尤为重要，有以下几个原因：

- 一个软件的生命周期中，80%的花费在于维护
- 几乎没有任何一个软件，在其整个生命周期中，均由最初的开发人员来维护
- 编码规范可以改善软件的可读性，可以让程序员尽快而彻底地理解新的代码
- 如果你将源码作为产品发布，就需要确任它是否被很好的打包并且清晰无误，一如你已构建的其它任何产品

为了执行规范，每个软件开发人员必须一致遵守编码规范。每个人。
```

#### 1.2命名规范
```
一个精准的命名能够让你的队友很容易的理解一个变量或者一个方法的用途,
如果一个不能表达其使用意义或者错误的命名会导致队友错误的理解，
就容易弄出bug或者更严重的错误,同样的道理类名也不能瞎胡命名,
良好的目录结构可以让队友很容易找到功能代码对应的位置，实现快速定位,
目前目录结构的区分按照功能模块，或者按照功能类型区分的都有，
目前我们的命名是按照 ‘功能类型’因为是微服务架构，实际上是已经先按功能模块做了大的微服务的分类了,
变量命令或者方法命名都是首字母小写+驼峰风格区分单词,
数据库命名还是通过下划线分割，url中的单词用中横线分割,JSON命名与java类命名一致
```


#### 1.3注释规范
```
注释这块不能偷懒,就算命名规范非常好的代码加上注释也不会给代码增加更多的可读难度，
更多的时候我们的命名规范不能够表达出具体的用意，加上注释，可以让队友更加精准的理解代码,
当然注释本身不能错误，不能为了应付检查把上一个方法的注释直接粘贴到下一个方法上,
这样的代码会误导队友，或者让队友看着代码很头大,当方法修改后，一定要记得修改注释，
不管什么原因造成的注释与实际情况不符的现象都是错误的
```


#### 1.4防御式编程与异常处理
```
防御式编程本身是在执行方法之前，对传入的参数加入更加严格的判断,
对于java本事定义一个新的异常是一件很容易的事情，
而异常也是java解决错误的一个比较好的方式，
但是要记得在接口出去的位置(controller advice)要加入友好的返回方式，
并且尽量将异常信息记录下来，用来以后核对错误原因,
对于方法内部尽量不要用try catch 语句,除非迫不得已,尽量将原始的异常让 
controller advice 截获到,并记录,这一切都是为了同时兼顾更好的核对问题，和友好的返回让客户可以接受
```


#### 1.5抽象出共用的代码部分
```
很多方法如果看上去一样的代码很多，其实是可以考虑抽象出一些公共的方法，
或者必须这样做，才是正确的，其实sping 框架 就是对web应用共性需求一个比较好的抽象,
它现在已经很强大,我们抽象针对的是具体的项目,
比如保存新闻对象这个方法，你不能在app 调用的时候写一遍 在后台写一遍，在h5或者网站部门再写一遍,
那样的代码，维护起来是很麻烦的，如果新加字段，或者修改逻辑，你需要修改很多处，
对于这样的问题，我们已经做了处理，那就是: 把业务逻辑尽量封装在servive层，
让其他controller 或者 web请求，只是做参数的接受,
你可以看到controller层很多地方代码很少,这样不仅可以降低工作的强度，
又可以很好的规划代码的结构，适应更多的变化,其实它能够降低你的bug率,
同样的功能再不同的代码中实现，很难保证逻辑的一致性，更容易发生bug
如何抽象代码是一个很深的东西,需要每个新程序员都多多学习
```

#### 1.6优化代码的结构
```
这里主要针对,判断语句或者循环语句比较多的情况,同样意义的代码，有人将代码包了很多很多层，
有人的代码确比较精简,
这里就是你需要巧妙的运用 return ,continue等语句,可以将很多层的代码简化,
if或者for 语句层数越多，队友看起来就会越头大，
或者过段时间自己都无法理解这些事情都容易出现
```


#### 1.7编码格式规范

```
/** 
 * Java编码格式个人推荐，参考JDK源码和Hyperic HQ源码（原spring旗下著名开源软件，现vmware）。 
 * @author lihzh(苦逼coder) 
 * 本文地址：http://mushiqianmeng.blog.51cto.com/3970029/737120
 */ 
public class CodeRule { 
     
    //声明变量，等号两边有空格。 
    private static int i = 1; 
     
    //方法声明，右括号和左大括号中间有空格。 
    public static void main(String[] args) { 
        //if语句，比较连接符(>)左右有空格，小括号和大括号中间有空格。 
        //if 与 左括号中间有空格 
        if (i > 0) { 
            System.out.println(i); 
        } 
        //两个条件的连接(&&)，左右有空格。 
        if (i > 0 && i < 2) { 
            System.out.println(i); 
        } 
         
        //if..else 语句两种格式 
        //1.参考JDK，个人使用方式，else跟大括号，前后都有空格 
        if (i > 0 && i < 2) { 
            System.out.println(i); 
        } else if (i > 2) { 
            System.out.println(i + 1); 
        } else { 
            System.out.println(i); 
        } 
        //2.参考Hyperic HQ源码, else另起一行，后仍有空格 
         if (i == 1) { 
             System.out.println(i); 
         } 
         else { 
             System.out.println(i); 
         } 
          
         //while语句，与if语句类型，while与括号中间有空格，括号内格式与if相同 
         while (i > 0 && i < 2) { 
             System.out.println(i); 
             i++; 
         } 
          
         //for语句，两种格式 
         //1.参考Hyperic HQ，个人使用方式。分号后带空格，每个子语句中，连接符左右都带空格。 
         //for与括号中间带空格，大小括号中间带空格。 
         for (int j = 0; j < 10; j++) { 
             System.out.println(i); 
         } 
         //2.参考JDK，区别在于子语句中，连接符左右无空格。 
         for (int j=0; j<10; j++) { 
             System.out.println(i); 
         } 
          
         //+-*/，格式，四则运算符号前后有空格。 
         //在JDK的有些代码里，在方法调用的参传递或在判断语句中存在的四则运算中，四则运算符号前后无空格。 
         //为了不造成困扰和混淆，个人为均保留空格。 
         int a = 1 + 2; 
         int b = 1 - 2; 
         int c = 1 * 2; 
         int d = 1 / 2; 
          
         //三元表达式格式，每个符号中间均有空格 
         int j = i > 2 ? 1 : -1; 
          
         //方法声明和调用，用逗号分隔的参数，逗号后有空格。 
         sum(a, b); 
         sum(c + d, j); 
    } 
     
    //方法声明，多个参数，逗号后有空格 
    private static int sum(int i, int j) { 
        return i + j; 
    } 
     
 
}
```
